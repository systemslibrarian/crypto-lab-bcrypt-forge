What bcrypt Actually Is

A Blowfish-based password hashing scheme designed by Niels Provos & David Mazières (1999). Deliberately slow and adaptive — bcrypt's cost factor lets security scale with hardware improvements.

Where bcrypt Is Used

  • Linux PAM (/etc/shadow)
  • PHP password_hash()
  • Node.js bcrypt / bcryptjs
  • Django (optional backend)
  • Ruby on Rails (default via has_secure_password)

Why Can't We Just Make It Faster?
The slowness IS the security. An attacker with a GPU farm faces the same cost per hash you do. Each cost increase doubles their work for every password in a breached database.

Why This Matters
A timing oracle attack recovers secrets one byte at a time by measuring comparison duration. Constant-time comparison is not optional — it is a fundamental requirement for any security-sensitive string comparison.

When Should You Choose bcrypt Over Argon2id?
When your stack defaults to it, or when FIPS compliance rules out Argon2. Otherwise prefer Argon2id for new systems.

Real-World Attack Demo

Three breach scenarios showing what happens when passwords are stored as plaintext, unsalted MD5, or bcrypt hashes.

Scenario A Plaintext Storage
Impact
Plaintext breach = immediate total compromise. Every user's password is exposed instantly.
Scenario B Unsalted MD5 Hashes
Impact
Rainbow tables precompute hashes for billions of common passwords. Unsalted hashes fall in seconds. Duplicate passwords produce identical hashes — one lookup compromises all users sharing that password.
Scenario C bcrypt Hashes (cost 12)
Impact
With bcrypt, a breached database buys your users time — potentially weeks or months — to change their passwords.
